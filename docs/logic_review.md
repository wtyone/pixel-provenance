# 代码逻辑与判断依据评估

这份文档记录 Pixel Provenance 当前判断逻辑的合理性、风险点和推荐解读方式。

## 总体结论

项目适合作为本地图片来源排查工具，但不适合作为自动鉴定器。合理的产品定位是“证据分层展示 + 人工复核”，不是“一键判断真假”。

当前较合理的部分：

- 把结构化元数据、字节关键词、修图痕迹、频域特征分成不同模块。
- 频域分析放在 Web Worker 中，避免阻塞页面。
- Adobe 规则没有使用泛化的 `Adobe` 字符串，避免 ICC 或普通修图误报。
- PNG 生成参数要求至少命中两个关键词，降低单个 `prompt` 字符串误报。
- 文件字节分布启发式被标记为弱信号，没有参与强 AI 命中。

需要保守解读的部分：

- C2PA/JUMBF 嗅探不是完整解析器，也不验证签名链。
- EXIF 相机字段可被后期写入，不能证明真实拍摄。
- 厂商关键词来自经验规则，仍可能漏报或误报。
- 频域评分没有训练集校准，阈值只适合做提示。
- 转换/导出功能会改变文件本身，因此导出后的元数据只能说明导出状态，不能说明原始来源。

## 证据优先级

1. 明确且可验证的来源凭证：例如经过签名链验证的 C2PA 声明。
2. 明确结构化声明：例如 `DigitalSourceType = trainedAlgorithmicMedia` 或 XMP/EXIF 直接记录生成工具。
3. 明确工具标记：例如 `ComfyUI`、`Midjourney`、`AdobeFirefly`、`SynthID` 等具体字符串。
4. 编辑软件痕迹：例如 Photoshop / Lightroom，只能证明处理链路，不等同于 AI。
5. 字节分布和频域异常：只作为弱线索，需要结合其他证据。

## 本次修正过的关键判断点

- 项目名已统一为 `pixel-provenance` / `Pixel Provenance`。
- C2PA 命中不再默认算作 AI 命中；只有 `DigitalSourceType` 明确属于算法生成类型时才进入 AI 类别。
- JUMBF 嗅探从“只要出现 `jumb` 字符串”改为先检查前置 box 长度是否合理，减少普通文本误报。
- 主流程复用已经解析出的元数据和 JUMBF 结果，避免重复解析。
- 结构化元数据的“是否存在元数据”改为基于实际可读字段，而不是只看少数提示字段。
- Google 规则移除了普通 `Google` 字符串，避免把 Google Pixel 真机 EXIF 误判为 Gemini / SynthID。
- Midjourney 规则移除了过短的 `midj` 字符串，降低随机字节误撞。
- 字节分布启发式增加小文件保护和正确采样计数，避免 NaN 或分母偏差。
- 频域特征排除了 DC 分量对低频比例和频谱熵的污染。
- 频域评分移除了“径向对称性高”规则，因为实值图像 FFT 幅度天然具有中心对称性，不能作为有效 AI 证据。
- 频域预处理从直接拉伸改为中心裁切后缩放，减少长宽比变形对方向性特征的影响。
- JPEG C2PA 清理不再删除所有 APP11 段，只处理看起来包含 C2PA/JUMBF 的段。
- 元数据面板不再把 `digitalCapture` 描述成已验证签名，因为当前工具没有做签名验证。

## 仍建议后续改进

- 接入真正的 C2PA 解析与签名验证库。
- 为检测规则增加测试样本：真实相机 JPEG、Google Pixel 照片、Photoshop 导出图、ComfyUI PNG、C2PA 样例、社交平台重编码图。
- 为频域评分建立小型标注集，统计误报率和漏报率后再调整阈值。
- 将频域判定文案完全国际化，目前 `src/frequency/panel.js` 仍有部分中文硬编码。
- 在 UI 中更清楚地区分“原图分析结果”和“导出文件分析结果”。

## 推荐结论模板

分析结果建议按下面方式表述：

> 当前文件中发现/未发现某类线索。该线索的强度为强/中/弱。它可以支持某种可能性，但不能单独证明图片来源。最终判断应结合原始文件、来源链路、签名验证和上下文材料。
